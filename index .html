1. CCS Installation and First Project

Before writing any code, I spent quite a bit of time just trying to get CCS working and create a project properly. The first confusion was after selecting an empty project template — there is a loading or buffering screen that appears. I kept waiting for it to do something, but the correct thing to do is click Import immediately without waiting for the buffer to finish.

The other thing that confused me was that most tutorials and YouTube videos show an older version of CCS called CCS Eclipse. I had CCS Theia installed, which has a completely different interface. The menu path in Theia for creating a project is: 

File → Create New Project...

Not File → New → CCS Project as most older tutorials show. These two lead to different screens and I wasted time trying to find options that simply didn't exist in my version.

One more thing — for LCD projects, the files hal_LCD.c and hal_LCD.h are not included in a blank project. I had to manually copy them from the Out-of-Box Experience project that comes with CCS.

2. Basic C Syntax Mistakes

Most of my early errors had nothing to do with the hardware. They were simple C syntax mistakes that I kept making because I was new to the language.

Missing # before include

// Wrong
include <msp430.h>

// Correct
#include <msp430.h>

Lowercase register and bit names

C is case-sensitive. Every register name like P1DIR and every bit name like BIT0 must be written in uppercase exactly as defined in the header file. Writing them in lowercase gives an undefined identifier error. 

// Wrong
p1dir |= bit0;
wdtctl = wdtpw | wdthold;

// Correct
P1DIR |= BIT0;
WDTCTL = WDTPW | WDTHOLD;

Missing semicolons

Every statement in C ends with a semicolon. I forgot this constantly in the beginning. The annoying part is the compiler sometimes points to the wrong line, making it hard to find. 

// Wrong
P1DIR |= BIT0
P1OUT |= BIT0

// Correct
P1DIR |= BIT0;
P1OUT |= BIT0;

3. Watchdog Timer Confusion

When I ran the blink LED code for the first time, the debugger started and a yellow arrow appeared on line 5. I thought something had gone wrong. It hadn't — that arrow just means the debugger paused at that line waiting for me to press Continue. I spent a few minutes looking for what the error was before realising there was no error at all. The Continue button is the triangle play button in the debug toolbar at the top left.

Separately, when I forgot to include the watchdog timer line, the LED did absolutely nothing. No error message, no warning — the chip was just silently resetting itself every 32 milliseconds because of the watchdog.

These two lines need to be at the top of every single program I write: 

WDTCTL  = WDTPW | WDTHOLD;   // stop watchdog timer — always line 1
PM5CTL0 &= ~LOCKLPM5;         // unlock GPIO pins    — always line 2
The second line, PM5CTL0 &= ~LOCKLPM5, is specific to FR-series chips. Without it, all GPIO pins are locked in high-impedance mode and produce no output even if the rest of the code is correct. This one caused me a lot of confusion before I found out about it.

4. Toggle LEDs with a Switch

When I tried to toggle between the red and green LEDs using the S1 button, I struggled specifically with where the while(1) loop goes and why the button logic felt backwards.

while(1) placed outside main

// Wrong — the while loop is outside main and never runs
void main(void)
{
    P1DIR |= BIT0;
}
while(1)
{
    P1OUT ^= BIT0;
}

// Correct — while loop is inside main
void main(void)
{
    P1DIR |= BIT0;
    while(1)
    {
        P1OUT ^= BIT0;
        __delay_cycles(500000);
    }
}

Button reads backwards — active-low logic

The button uses a pull-up resistor, which means the pin reads HIGH (1) when the button is not pressed and LOW (0) when it is pressed. This is the opposite of what I expected. So to detect a press, the condition needs the NOT operator: 

// Wrong — this is true when button is RELEASED
if (P1IN & BIT2)

// Correct — this is true when button is PRESSED
if (!(P1IN & BIT2))

The full button setup needs three lines:

P1DIR &= ~BIT2;   // set as input
P1REN |= BIT2;    // enable internal resistor
P1OUT |= BIT2;    // configure as pull-up

5. LCD Interface and Display Issues

The LCD was the most difficult part of the whole learning process. I hit several different problems one after another.

Problem 1 — Undefined register names

My first LCD attempts used examples copied from other MSP430 devices. They failed to compile because register names like P5SEL1, LCDMUX_3, VLCD_30, LCDCLR, and LCDPRE_5 simply do not exist on the FR4133. The fix was to stop using other device examples and instead use the hal_LCD.c and hal_LCD.h files from the FR4133 Out-of-Box Experience project. After that, everything came down to three functions: 

Init_LCD();
clearLCD();
showChar('A', pos1);

Problem 2 — Digits fading, last digit invisible

After getting the LCD working, pressing the button showed digits 1 to 6 but they faded from left to right. Digit 6 was barely visible or completely gone. I initially thought it was a hardware issue with the LCD itself.

The actual cause was that clearLCD() was being called every single loop pass. The loop runs thousands of times per second, so the display was being wiped before it had time to fully show anything. Digit 6 was especially bad because it was the last one written before the loop restarted.

The fix was to only update the LCD when the button state actually changes:

unsigned char lastState = 0;

while(1)
{
    if (!(P1IN & BIT2))
    {
        if (lastState != 1)
        {
            clearLCD();
            showChar('1', pos1);
            showChar('2', pos2);
            showChar('3', pos3);
            showChar('4', pos4);
            showChar('5', pos5);
            showChar('6', pos6);
            lastState = 1;
        }
    }
    else
    {
        if (lastState != 0)
        {
            clearLCD();
            lastState = 0;
        }
    }
}

6. LCD Character Display Issues

Some characters on the LCD looked distorted or not quite right. I also got undefined identifier errors a few times when using position names.

The FR4133 LCD is a 7-segment display, not a pixel screen. Characters are made up of fixed segment combinations defined inside hal_LCD.c. Some letters just do not look great on a 7-segment display — that is a hardware limitation, not a bug.

The position constants pos1 through pos6 are defined in hal_LCD.h and must be written exactly that way. Wrong casing causes a compile error.

7. Interrupt Register Order

When learning about GPIO interrupts, my program either fired a spurious interrupt at startup without any button press, or it never responded to button presses at all. The problem was the order in which I configured the interrupt registers.

If P1IE is enabled before P1IFG is cleared, any stale flag already sitting in the register will trigger an interrupt immediately. The correct order is: 

P1IES |= BIT2;          // 1. choose edge — falling means button press
P1IFG &= ~BIT2;         // 2. clear any existing stale flag
P1IE  |= BIT2;          // 3. enable the interrupt
__bis_SR_register(GIE); // 4. global interrupt enable
RegisterWhat it does
P1IESChoose which edge triggers — falling (1 to 0) or rising (0 to 1)
P1IFGFlag — gets set when the event happens. Must be cleared manually.
P1IEEnables or disables the interrupt on that pin

8. Potentiometer and ADC Configuration

I connected a potentiometer to P1.3 and tried to display the reading on the LCD. The code compiled and ran, but the output only went from 0 to 255 instead of the expected range.

Wrong resolution constant

I was using ADCRES_2 which does not exist on the FR4133. The chip silently fell back to 8-bit mode with no error or warning. 

// Wrong — ADCRES_2 does not exist on FR4133
ADCCTL2 = ADCRES_2;

// Correct — use ADCRES1 for 12-bit on FR4133
ADCCTL2 = ADCRES1;    // gives range 0 to 4095

P1SEL1 undefined error

// Wrong — P1SEL1 does not exist on FR4133
P1SEL0 |= BIT3;
P1SEL1 |= BIT3;

// Correct — P1SEL0 alone is enough
P1SEL0 |= BIT3;

ADC configuration being ignored

Some configuration changes were not taking effect. The MSP430 requires ADCENC to be cleared before any ADC registers can be written. If it is already set, the chip silently ignores the writes. 

ADCCTL0 &= ~ADCENC;          // must disable before configuring
ADCCTL0  = ADCSHT_2 | ADCON;
ADCCTL1  = ADCSHP;
ADCCTL2  = ADCRES1;
ADCMCTL0 = ADCINCH_3;

9. HC-SR04 Ultrasonic Sensor

This was my first attempt at connecting an external sensor. Several problems appeared one after another.

Problem 1 — Red LED faintly blinking, garbage readings

The readings were completely wrong and the red LED was faintly flickering in the background. The timer was not running at the expected speed because the clock was not configured explicitly. I had assumed a default frequency which turned out to be wrong. The fix was to set the clock explicitly before any timer code: 

__bis_SR_register(SCG0);
CSCTL0 = 0;
CSCTL1 = DCOFSEL_0;
CSCTL2 = SELA__VLOCLK | SELM__DCOCLK | SELS__DCOCLK;
CSCTL3 = DIVA__1 | DIVM__1 | DIVS__1;
__bic_SR_register(SCG0);

Problem 2 — Echo pin stuck HIGH constantly

After fixing the clock the echo pin was still reading HIGH all the time. I tried adjusting delays and adding debug output but nothing changed. The actual cause turned out to be the wiring.

Problem 3 — Wires connected to wrong pins (root cause)

The TRIG and ECHO wires were physically plugged into different pins than what the code was using. The echo pin was floating — not connected to anything — so it read HIGH randomly. Once I matched the physical wires to the pin numbers in the code, the sensor worked correctly. 

// Code defined these pins — physical wires must match
#define TRIG_PIN  BIT4    // P1.4
#define ECHO_PIN  BIT5    // P1.5
When sensor output is completely wrong, check the physical wiring before changing any code. This saved me a lot of time once I learned it.

Problem 4 — Last two LCD digits showing all segments lit

Positions pos5 and pos6 were showing a fully-lit block character instead of being blank. They had never been written to, so they held leftover values in LCD memory. The fix was to always write all six positions and use a space character for the unused ones. 

showChar(' ', pos5);
showChar(' ', pos6);

// If the space character does not fully blank them
LCDM10 = 0x00;
LCDM12 = 0x00;
The all-segments issue on pos5 and pos6 is still being looked into. The space character fix works most of the time.

10. MPU6050 Accelerometer via I2C

This was my first time working with I2C on the MSP430. The MPU6050 module was labeled HW-290 and I initially assumed it was an analog ADXL335 accelerometer. After reading the markings on the actual chip I found it was an MPU6050 — a digital I2C sensor with both a 3-axis accelerometer and a 3-axis gyroscope.

Problem 1 — I2C register names undefined

Compilation failed because register names I found in examples were not available on the FR4133. The FR4133 uses eUSCI_B0 for I2C and the register names are different from what many online examples show. 

// Direct register access needed for FR4133
#define UCB1CTL1_REG  (*((volatile unsigned int *) 0x0640))
#define UCB1BR0       (*((volatile unsigned int *) 0x0644))
#define UCB1IFG_R     (*((volatile unsigned int *) 0x066C))

Problem 2 — Sensor always returns X=0, Y=0, Z=0

Even after getting I2C communication working, all three axes returned zero. The MPU6050 boots in sleep mode by default and will not return any data until it is explicitly woken up. 

// Wake the sensor before reading anything
I2C_WriteReg(MPU_ADDR, PWR_MGMT_1, 0x00);
__delay_cycles(100000);

Problem 3 — SDA and SCL connected to wrong pins

The sensor did not respond even after fixing the code. I had connected the wires to pins labelled I2C on the board silkscreen, but the software was configured to use eUSCI_B0 which maps to different physical pins. Checking the actual datasheet pinout diagram fixed this. 

// FR4133 eUSCI_B0 I2C pins — verify from datasheet, not silkscreen
// P1.6 = UCB0SDA
// P1.7 = UCB0SCL
On MSP430 devices, a pin can serve multiple functions depending on how it is configured in software. The same pins that work for SPI can also work for I2C if the eUSCI module is configured differently. The board silkscreen only shows one label per pin so it can be misleading.

11. microSD Card via SPI

The SD card interface was one of the more involved exercises. Multiple things went wrong before it worked.

Problem 1 — Wrong SPI pins

The card did not respond at all during initialization. I had assumed pin locations without checking the datasheet first. After verifying against the FR4133 pinout: 

P5.0 → CS
P5.1 → SCLK
P5.2 → MOSI
P5.3 → MISO

Problem 2 — SPI peripheral mismatch

After fixing the pins, communication still failed. The initialization code was setting up UCB1 but the data transfer functions were using UCB0. These are two different hardware peripherals so the data never actually went anywhere. Making sure both used the same peripheral fixed it.

Problem 3 — Program hanging inside SPI transfer

The LCD showed nothing and the program stopped responding. The code was waiting inside the SPI transfer function for a response that never came, caused by the peripheral mismatch above.

Problem 4 — CMD0 not responding, LCD showing FAIL or RSP00

After fixing the peripheral mismatch, the SD card still failed initialization. Several things were investigated — SPI pin config, peripheral selection, startup timing, and supply voltage. The SD module was being powered from 3.3V. When I switched it to 5V, CMD0 responded correctly and initialization completed. 

// SD card module VCC → 5V pin on LaunchPad
// Many SD adapter modules have onboard regulators designed for 5V input
// The SD card itself runs at 3.3V internally but the module may need 5V
MSP430 GPIO pins are not 5V tolerant. When the SD module runs on 5V, the MISO line can send 5V signals back to the MSP430. A voltage divider is needed on MISO to bring it down to a safe level. 
SD MISO (5V) → 1kΩ → MSP430 MISO pin
                   ↓
                  2kΩ
                   ↓
                  GND
// Result: ~3.3V safe for MSP430

Problem 5 — SD card needs dummy clocks at startup

Even with correct power and pins, initialization was unreliable. SD cards require at least 74 dummy clock pulses before CMD0 to properly enter SPI mode. 

for(int i = 0; i < 10; i++)
    SPI_Transfer(0xFF);   // 10 x 8 bits = 80 clock pulses

Memory constraint — raw write instead of FAT

The FR4133 has only 2KB of RAM. A full FAT file system implementation needs a 512-byte sector buffer plus overhead, which risks stack overflow on this chip. I chose a simple raw write approach instead. The data format used was: 

press_count
1,1
2,2
3,3
...

This opens directly in Excel without any conversion needed.

12. Button Counter Project

This was the final project combining everything learned up to this point. Pressing S1 increments a counter, displays the current count on the LCD, and writes each press to the SD card immediately.

The SD stuck or RSP00 issue that appeared during the SD card exercise was always caused by the supply voltage. After making sure the SD module was on 5V, the CMD0 response was reliable.

Data saved to card:

press_count
1,1
2,2
3,3
4,4

13. Key Lessons Learned

These are the things that came up repeatedly and took me a while to understand properly.

Always verify pins from the datasheet, not the silkscreen

The labels printed on the board are a rough guide only. For any peripheral — SPI, I2C, UART — I always cross-checked the actual pin function using the MSP430FR4133 datasheet and the LaunchPad schematic. This saved a lot of debugging time once I made it a habit.

Pin multiplexing — one pin, multiple functions

The same physical pin can be SPI or I2C depending on how the eUSCI module is configured in software. This caught me off guard a few times. 

P5.2 → UCB0SIMO  (when configured as SPI)
P5.2 → UCB0SDA   (when configured as I2C)

Voltage levels matter

ComponentSupplySafe for MSP430?
MPU60503.3VYes
HC-SR045VLevel shifting needed on ECHO
SD card module5VVoltage divider needed on MISO

Check hardware before rewriting code

Multiple times I spent hours adjusting code when the actual problem was a loose wire or a wire in the wrong hole. Now I always verify the physical connections first if something is completely not working.

Test each part separately

Instead of building the full system and testing it all at once, verifying each piece independently made debugging much faster. Sensor first, then communication, then display, then storage, then integration.

14. Master Code Skeleton

Every program I wrote started with this structure:

#include <msp430.h>

void main(void)
{
    WDTCTL  = WDTPW | WDTHOLD;   // 1. stop watchdog   — always first
    PM5CTL0 &= ~LOCKLPM5;        // 2. unlock GPIO pins — always second

    // 3. configure pins
    //    output:  PxDIR |= BITx
    //    input:   PxDIR &= ~BITx;  PxREN |= BITx;  PxOUT |= BITx
    //    analog:  PxSEL0 |= BITx

    // 4. configure peripherals

    while(1)
    {
        // 5. main logic
    }
}
